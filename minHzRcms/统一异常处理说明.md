# 统一异常处理说明

## 概述

根据用户要求，实现了统一的API请求异常处理机制，避免在各个页面重复显示错误提示。

## 实现方案

### 1. API层面统一处理

在 `utils/api.js` 的 `requestWithRetry` 函数中，当所有重试都失败后，显示统一的错误提示：

```javascript
fail: (error) => {
  console.error(`❌ API请求失败 (尝试 ${retryCount + 1}):`, error)
  if (retryCount < maxRetries - 1) {
    console.log(`🔄 重试请求 (${retryCount + 2}/${maxRetries})...`)
    setTimeout(() => attempt(retryCount + 1), 1000 * (retryCount + 1))
  } else {
    console.error(`💥 所有重试都失败了，最终错误:`, error)
    // 显示统一的错误提示
    wx.showToast({
      title: '网络请求失败，请稍后重试',
      icon: 'error',
      duration: 2000
    })
    reject(error)
  }
}
```

### 2. 移除页面中的重复错误提示

移除了以下页面中的重复错误提示：

#### pages/report/report.js
- 移除删除报告失败时的错误提示
- 保留删除成功的提示（用户操作反馈）

#### pages/create-report/create-report.js
- 移除提交报告失败时的错误提示
- 保留提交成功的提示（用户操作反馈）

#### pages/report-detail/report-detail.js
- 移除获取报告详情失败时的错误提示

## 优势

### 1. 统一用户体验
- 所有API请求失败都显示相同的错误提示
- 避免不同页面显示不同的错误信息

### 2. 减少代码重复
- 不需要在每个页面的 `.catch()` 中重复写错误提示
- 代码更简洁，维护更容易

### 3. 集中管理
- 错误处理逻辑集中在API层面
- 便于后续调整错误提示内容

## 保留的提示

### 1. 成功操作提示
- 删除报告成功
- 提交报告成功
- 更新报告成功

### 2. 业务逻辑错误
- 报告不存在
- 页面跳转失败
- 表单验证失败

### 3. 用户确认对话框
- 删除确认对话框

## 错误处理流程

1. **API请求发起** → 自动重试机制
2. **重试失败** → 显示统一错误提示
3. **页面接收错误** → 记录日志，执行备用逻辑（如使用模拟数据）

## 测试建议

1. **网络异常测试**：断开网络，验证统一错误提示
2. **服务器异常测试**：关闭服务器，验证错误处理
3. **功能测试**：确保所有功能正常工作
4. **用户体验测试**：验证错误提示是否友好

## 注意事项

1. **错误日志**：所有错误仍然会记录到控制台，便于调试
2. **备用逻辑**：页面中的备用逻辑（如模拟数据）仍然保留
3. **成功提示**：重要的成功操作提示得到保留
4. **向后兼容**：修改不影响现有功能逻辑
